# fast-uni-app-v3-js 脚手架审查与优化建议

## 1. 结论

当前项目已经具备一个 uni-app 基础模板的雏形，主要完成了这些事情：

- 已接入 `uni-app + Vue3 + Pinia`
- 已挂载 `uv-ui`、`jasper-ui`
- 已有请求封装、Mock 示例、Pinia 示例、时间工具和加密工具
- 已补充了一批说明文档，适合团队内部参考

但从“可复用脚手架”的标准来看，现在更像是一个 **可运行的演示模板**，还不是一个 **工程化完整、适合多人长期复用的新项目底座**。

核心问题不在页面能不能跑，而在于下面几类基础设施还没有真正闭环：

- 工程脚本缺失
- 代码规范缺失
- Mock / PostCSS 只有依赖和文档，缺少实际接线
- 登录鉴权 / 路由守卫 / 全局异常处理还是占位实现
- 环境配置与 README 还不够适合新项目直接复制使用

## 2. 现状审查

### 2.1 已有能力

- 入口已完成插件挂载，见 `main.js`
- 请求层已抽出统一配置和拦截器，见 `util/request/index.js`
- 有 Pinia 示例 store，见 `stores/sample.js`
- 有 Mock 数据和演示页，见 `mock/mock.js`、`api/mock/mock.js`
- 有基础文档目录，见 `docs/`

### 2.2 当前最明显的问题

#### A. 缺少脚手架最基本的工程命令

`package.json` 里只有一个占位 `test` 命令：

- `package.json:6-8`

实际验证结果：

- 执行 `pnpm test` 后返回 `Error: no test specified`

这意味着项目目前没有形成标准化的开发入口，例如：

- 本地开发命令
- 构建命令
- lint 命令
- format 命令
- 文档生成或校验命令

对于脚手架来说，这是最高优先级缺口之一。

#### B. Mock / PostCSS 依赖已安装，但仓库内缺少实际配置文件

当前依赖里声明了：

- `autoprefixer`
- `mockjs`
- `vite-plugin-mock`
- `postcss-px-to-viewport`

见：

- `package.json:15-19`

但仓库里没有看到实际的 `vite.config.js/ts`、`postcss.config.js` 等入口配置文件。当前状态更像“准备好了说明文档”，而不是“已经集成完成”。

这会带来两个问题：

- 新成员会以为 Mock 和 PostCSS 已经可直接使用
- README 与真实项目状态存在认知偏差

#### C. 请求层还停留在演示级，不适合作为生产模板直接复用

请求封装已存在，但还有几个明显缺口：

1. token 注入还是注释占位
   - `util/request/requestInterceptors.js:11-15`

2. 失败分支里返回 pending promise，会导致调用方悬空
   - `util/request/responseInterceptors.js:28-31`

3. 第三方域名透传判断不够清晰
   - `util/request/responseInterceptors.js:11-12`
   - 但示例里实际是通过 `baseURL` 直接传入，如 `api/mock/mock.js:19`、`api/mock/mock.js:27`

4. 统一错误模型不稳定
   - 成功分支按 `data.code === 200` 判断
   - 但 mock 示例里又存在 `code: 0` 的返回结构，见 `mock/mock.js`

这说明当前请求层能演示“怎么发请求”，但还没有统一好“项目以后都按什么规范收口”。

#### D. 路由拦截器存在，但尚未形成实际能力

`util/interceptors/uni-route-interceptor.js` 里只有通用拦截框架，没有真正的白名单、登录校验、权限跳转、重复跳转控制等逻辑：

- `util/interceptors/uni-route-interceptor.js:1-33`

而且当前入口里也没有看到它被统一注册到应用生命周期中。对于脚手架来说，这部分通常应该至少提供：

- 登录白名单
- 未登录跳转登录页
- 已登录禁止进入登录页
- 页面埋点或路由日志的预留位置

#### E. 示例内容较多，但“业务骨架”不足

现在项目里有较多演示性质内容：

- 首页演示 Pinia、请求、加密、时间工具
- 用户页演示 Mock

但缺少基础业务模块骨架，例如：

- 登录态管理
- 用户信息管理
- 存储封装
- 上传封装
- 分页列表基类
- 表单校验策略
- 权限页面 / 404 / 无网络 / 空状态页面

这会导致项目更像“功能拼盘示例”，而不是“新业务可以直接往上填”的模板。

#### F. 工程规范与协作规范缺失

当前仓库没有看到这些基础设施：

- ESLint
- Prettier
- EditorConfig
- Husky
- lint-staged
- commitlint
- CI 配置

同时 `.gitignore` 也比较精简，仅忽略了几个目录：

- `.gitignore:1-4`

对于要复用的脚手架，这些规范项通常应该默认具备，否则团队协作成本会持续上升。

## 3. 建议的优化优先级

## 3.1 P0：建议优先补齐

这些内容建议优先处理，因为它们决定这个项目能不能称为“脚手架”。

### 1. 补全工程命令

建议至少补这几类脚本：

- `dev:h5`
- `build:h5`
- `lint`
- `lint:fix`
- `format`
- `test`

如果当前项目主要依赖 HBuilderX，也建议至少在 README 中明确：

- 哪些操作必须走 HBuilderX
- 哪些操作可以走命令行

否则脚手架的可迁移性和自动化能力都很弱。

### 2. 真正接入 Mock / Vite / PostCSS

建议把“文档示例”变成“仓库默认配置”：

- 新增 `vite.config.js` 或 `vite.config.ts`
- 接入 `vite-plugin-mock`
- 挂载 `autoprefixer`
- 挂载 `postcss-px-to-viewport` 或明确说明暂不启用

如果不打算默认启用，也建议删掉依赖或在 README 明确标注“仅文档预留，未默认接入”。

### 3. 固化请求层规范

建议统一约定：

- 接口成功码标准
- `code / message / data` 字段标准
- 业务异常与网络异常的处理方式
- token 注入位置
- token 失效后的统一处理
- 第三方域名请求的调用方式
- 是否允许拦截器内吞错

尤其建议去掉 `new Promise(() => {})` 这种挂起写法，统一改为显式 `reject` 或标准返回体，避免页面请求无响应。

### 4. 提供最小可用的登录骨架

脚手架不一定要带完整账号体系，但建议至少具备：

- `stores/modules/user.js` 或等效用户 store
- token 存取封装
- 登录态判断工具
- 路由白名单
- 未登录跳转逻辑

这会显著提升模板的复用价值。

### 5. 补齐代码规范链路

建议至少增加：

- `ESLint`
- `Prettier`
- `.editorconfig`
- `husky + lint-staged`
- `commitlint`

这是基础脚手架最有性价比的一组投入，能立刻改善代码一致性。

## 3.2 P1：建议作为第二阶段补齐

### 1. 重构目录为更可扩展的业务结构

当前目录已经能用，但从长期维护看，建议考虑往下面这种方向整理：

- `api/modules`
- `stores/modules`
- `composables`
- `config`
- `constants`
- `utils`
- `services`
- `guards`

这样比把所有能力散落在 `common`、`util`、`api` 下更容易扩展。

### 2. 增加通用基础能力封装

脚手架建议补齐这些通用模块：

- 本地存储封装
- 上传文件/图片封装
- 下载与预览封装
- 分页查询基类
- 防抖节流工具
- 表单校验工具
- 设备信息与平台判断工具
- 日志上报接口封装

### 3. 提供基础业务页面模板

建议新增最少一组通用页面模板：

- 登录页
- WebView 页
- 列表页
- 详情页
- 表单页
- 空状态页
- 错误页

脚手架的价值不只是在“组件装好了”，更在于“新页面有可复制的标准结构”。

### 4. 完善环境配置策略

建议新增：

- `.env.example`
- 环境变量说明表
- 不同环境的命名规范
- 接口域名、静态资源域名、开关项说明

现在仓库里有多个 `.env.*`，但缺少一个给新项目初始化使用的标准模板。

## 3.3 P2：可选增强项

### 1. 测试能力

如果团队对稳定性有要求，建议逐步补：

- 单元测试
- 请求层测试
- 工具函数测试

不一定一开始就覆盖页面层，但至少应该让工具函数和请求层可验证。

### 2. 自动化与发布

建议后续补充：

- CI 校验
- 自动 lint
- 自动测试
- 自动发版或变更日志生成

### 3. TypeScript 版本分支

当前项目是 JS 模板。如果未来要长期作为基础模板维护，建议考虑：

- 保留当前 JS 版
- 另提供 TS 版脚手架

因为不少团队会直接要求 TS 作为默认基线。

## 4. 作为基础脚手架，建议默认具备的内容清单

下面是一份更偏“标准答案”的脚手架基线清单。

### 工程层

- 标准开发 / 构建 / 校验脚本
- ESLint / Prettier / EditorConfig
- commitlint / husky / lint-staged
- `.env.example`
- 完整 `.gitignore`
- README 初始化说明
- 更新日志策略

### 运行时层

- 请求封装
- 响应码统一处理
- 全局异常提示
- 登录态管理
- 路由守卫
- 本地缓存封装
- 平台差异处理

### 业务层

- 用户模块
- 分页列表示例
- 表单页示例
- 上传示例
- 鉴权页示例
- 空态 / 错误态页面

### 文档层

- 快速开始
- 目录结构说明
- 环境变量说明
- 接口规范说明
- Mock 使用说明
- 发版说明
- 常见问题

## 5. 推荐落地顺序

如果要控制投入，我建议按下面顺序推进：

1. 先补 `package.json` 脚本、Vite 配置、Mock 配置、PostCSS 配置
2. 再补 ESLint / Prettier / Husky / Commitlint
3. 然后重构请求层，统一接口返回和错误处理
4. 再补用户 store、token 管理、路由白名单
5. 最后再整理示例页面和业务模板页

这样做的收益最大，也最接近“把演示项目升级成真正脚手架”。

## 6. 一句话判断

这个项目目前适合：

- 作为 uni-app 新项目的参考起点
- 作为内部示例模板

但还不太适合：

- 直接作为团队统一脚手架长期复用
- 直接交给新成员无额外补充就开始正式业务开发

如果只做一轮优化，建议优先把“工程命令、规范链路、请求层、登录骨架、Mock 真正接线”这五件事补齐，项目的脚手架完成度会明显提升。
