# jasper-login 组件优化建议

## 1. 结论

这个组件有明显优化空间，而且不是“锦上添花”级别，而是已经涉及到可维护性、可移植性和上线稳定性的问题。

从当前目录实现看，组件已经具备统一登录入口的基础能力，但还存在以下典型现状：

- 页面职责混杂，UI、平台判断、路由跳转、登录编排、登录后持久化写在一起。
- 示例代码与真实业务代码耦合较深，组件还不够“可插拔”。
- 事件总线、外部依赖、平台兼容分支较多，后续改动容易引入隐性回归。
- 部分文件存在中文乱码，说明仓库在编码、发布或协作流程上存在基础治理问题。
- 还有空壳文件和缺失依赖，说明组件完整度仍可继续打磨。

建议把本组件的优化目标定义为：

1. 从“项目内可用的登录页面集合”升级为“可配置、可维护、可复用的统一登录模块”。
2. 从“示例驱动”升级为“接口适配驱动”。
3. 从“页面事件拼接”升级为“登录流程编排 + 状态收敛”。

## 2. 当前代码里比较明确的问题

### 2.1 存在空壳组件，模块出口不完整

以下文件目前基本为空：

- `components/jasper-login/jasper-login.vue`
- `pages/login/login-withpwd.vue`

这会带来两个问题：

- 模块对外主组件语义不清，使用方不知道应该直接用哪个入口。
- “账号密码登录”在配置层已出现，但页面实际未实现，容易让调用方误判能力边界。

建议：

- 明确 `jasper-login.vue` 是否作为统一对外入口。
- 如果短期不支持账号密码登录，就先从 `config.js` 和说明文档中移除该能力描述。
- 如果要支持，则优先补齐账号密码登录页和对应 API 适配层。

### 2.2 组件自包含性不足，依赖了宿主项目实现

在 `components/b-fab-login/b-fab-login.vue` 中存在：

- `import {getPhoneInfoByCode} from '@/api';`

同时引用了：

- `@/uni_modules/jasper-login/common/weixin-mixin.js`

但当前模块目录内并没有发现 `common/weixin-mixin.js` 文件。

这意味着当前组件并不是一个真正自包含的 uni_modules 组件，而是默认依赖宿主项目已有实现。这样会导致：

- 组件迁移到新项目时无法直接工作。
- 使用者需要猜测哪些接口必须额外补齐。
- 文档说明与实际依赖不一致。

建议：

- 把所有对外依赖收敛到 `adapters/` 或 `services/` 目录，例如：
  - `services/auth.js`
  - `services/sms.js`
  - `services/weixin.js`
- 通过配置注入或适配器注入的方式接入宿主项目 API，不要直接写死 `@/api`。
- 缺失的 `weixin-mixin.js` 要么补齐，要么取消 mixin 方案，改成组合式/工具函数。

### 2.3 示例 API 与真实登录流程耦合过深

`common/sampleApi.js` 目前不仅是示例，还直接承担登录流程中的关键职责：

- 发送短信验证码
- 校验验证码
- 登录并返回 token

这会带来两个问题：

- 使用方容易把示例代码误当生产实现。
- 组件内部流程已经绑定了返回结构，不利于替换后端。

建议：

- 将 `sampleApi.js` 明确降级为 `examples/sample-auth-api.js`。
- 组件内部只依赖统一接口协议，例如：
  - `sendSms(phoneNumber)`
  - `verifySms(phoneNumber, code)`
  - `loginByPhone(payload)`
  - `loginByProvider(payload)`
- 把返回值统一成稳定结构，例如：

```js
{
  success: true,
  token: '',
  tokenExpired: '',
  userInfo: {},
  accountId: ''
}
```

### 2.4 登录页承担了过多业务逻辑

`pages/login/login-withoutpwd.vue` 当前同时负责：

- 登录方式选择
- 手机号输入与格式化
- 短信验证码发送
- 微信手机号授权
- 登录成功后的用户信息持久化
- 返回逻辑
- 事件监听与派发

这会让页面变成“超级页面”，后续任何一个登录方式变更都容易影响其它流程。

建议拆分：

- `composables/useLoginFlow.js` 或 `common/use-login-flow.js`
  - 负责登录编排
- `composables/useSmsLogin.js`
  - 负责短信验证码能力
- `composables/useThirdPartyLogin.js`
  - 负责微信、一键登录、Apple 等平台能力
- 页面只保留：
  - 表单展示
  - 用户交互
  - 调用 composable/service

### 2.5 事件总线使用较重，且清理不完全

当前实现大量使用：

- `uni.$on`
- `uni.$emit`
- `uni.$off`

这类全局事件总线在登录链路里虽然方便，但问题是：

- 事件名是字符串，缺乏约束。
- 页面跳转后容易出现残留监听。
- 排查问题时链路不透明。

从代码看，`login-withoutpwd.vue` 的卸载清理也不完全一致：

- `onLoad` 里注册的是 `login-setLoginType`、`login-get-smscode-event`、`login-api-result-event`
- `onUnload` 里却只写了 `uni.$off('uni-id-pages-setLoginType')`
- 实际清理逻辑更多写在 `methods.unLoad()`，但这个方法名不是页面生命周期，不会自动执行

这属于明确的稳定性风险。

建议：

- 优先改为页面内显式回调传递，减少事件总线。
- 如果必须保留事件总线，至少统一事件常量：

```js
export const LOGIN_EVENTS = {
  SET_TYPE: 'login-setLoginType',
  GET_SMS_CODE: 'login-get-smscode-event',
  API_RESULT: 'login-api-result-event'
}
```

- 所有监听必须在 `onUnload` 中一一对应解除。
- 删除伪生命周期方法 `unLoad()`，避免误导维护者。

### 2.6 编码异常问题比较严重

当前多个文件存在明显中文乱码，例如：

- `readme.md`
- `config.js`
- `components/b-fab-login/b-fab-login.vue`
- `pages/login/login-withoutpwd.vue`
- `common/login-page.scss`

这不是显示问题，而是代码资产质量问题，会直接影响：

- 二次维护
- 代码评审
- 开源发布
- IDE 搜索和国际化整理

建议优先处理：

- 统一项目文件编码为 `UTF-8`
- 检查 Git 属性和编辑器保存配置
- 对 README、注释、页面文案做一次完整修复

这一项建议列为 P0。

### 2.7 配置项存在重复、边界不清的问题

`config.js` 中 `loginTypes` 里 `univerify` 出现了重复定义。

这说明目前配置层还没有形成稳定约束，后续容易出现：

- 登录方式顺序异常
- 重复渲染
- 平台条件分支互相覆盖

建议：

- 在配置初始化时做去重。
- 将配置分为三层：
  - `base config`
  - `platform overrides`
  - `runtime normalization`

例如：

```js
export function normalizeLoginConfig(config) {
  return {
    ...config,
    loginTypes: [...new Set(config.loginTypes || [])]
  }
}
```

### 2.8 登录后数据持久化强依赖全局对象

`handleLoginApiRes` 中直接依赖：

- `uni.$unc.userStore`
- `uni.$unc.getValue("tokenKey")`
- `uni.$constants.RESULT_SUCCESS_CODE`

这会让组件和 `jasper-ui`、宿主项目全局注入强耦合。

建议：

- 把“登录成功后的数据落库/落缓存”抽成可配置 hook：
  - `onLoginSuccess`
  - `persistToken`
  - `persistUserInfo`
- 给默认实现，但允许宿主替换。

这样模块会更像一个可集成组件，而不是固定绑定某个项目脚手架。

### 2.9 一键登录和第三方登录流程里还有演示代码残留

`b-fab-login.vue` 中 `univerify()` 成功后直接写死：

```js
let phoneNumber = 16270569395
this.login_After(phoneNumber, 'univerify')
```

这属于明显的演示残留，不适合保留在组件主流程中。即便注释里写了说明，也会造成：

- 调试与生产逻辑混淆
- 集成方误判流程已经打通

建议：

- 生产链路只保留真实云函数/服务端调用。
- 演示逻辑放入 `mock` 开关中，且默认关闭。

### 2.10 命名与结构还有统一空间

当前目录同时存在：

- `b-fab-login`
- `b-agreements`
- `jasper-login`
- `sampleApi`
- `login-withoutpwd`

整体命名风格不完全统一，组件、页面、工具、示例文件混杂。

建议统一命名规则，例如：

- 组件：`jasper-login-panel.vue`、`jasper-login-methods.vue`
- 页面：`login-phone.vue`、`login-code.vue`
- 服务：`auth.service.js`
- 配置：`login.config.js`

统一命名后，后续扩展如“游客登录”“绑定手机号”“注销后重新登录”会更顺畅。

## 3. 优化优先级建议

### P0：必须优先处理

- 修复全仓 UTF-8 编码问题和乱码文案。
- 清理空壳文件或补齐实现。
- 修复事件总线注册/解绑不对称问题。
- 去掉主流程中的演示残留代码。
- 补齐缺失依赖说明，尤其是 `weixin-mixin.js` 和 `@/api` 依赖。

### P1：建议尽快处理

- 抽离登录服务层和接口适配层。
- 将登录成功后的缓存/状态写入改为可配置 hook。
- 对 `config.js` 做配置归一化、去重和平台裁剪。
- 拆分页面中的大段流程逻辑，降低单文件复杂度。

### P2：中期优化

- 补账号密码登录。
- 增加类型声明，至少给关键接口补 JSDoc 或 TypeScript 类型。
- 增加最小可运行示例和接入文档。
- 为每种登录方式增加失败态、降级态和埋点。

## 4. 推荐的重构方向

建议把结构调整为下面这种更清晰的形式：

```text
jasper-login/
  components/
    login-panel.vue
    login-method-list.vue
    agreements.vue
  pages/
    login-phone.vue
    login-code.vue
    login-password.vue
  services/
    auth.service.js
    sms.service.js
    provider.service.js
  composables/
    use-login-flow.js
    use-sms-login.js
    use-provider-login.js
  constants/
    events.js
    login-types.js
  utils/
    normalize-config.js
    route.js
  adapters/
    sample-adapter.js
  config/
    default.js
```

核心原则：

- 页面只做交互。
- service 只做接口请求。
- composable 只做流程编排。
- adapter 负责对接宿主项目差异。
- config 负责开关和平台能力描述。

## 5. 文档层面的建议

当前 `readme.md` 本身已经出现乱码，建议顺手一起重写。至少补齐以下内容：

- 组件支持哪些登录方式。
- 哪些能力是示例，哪些能力可直接用于生产。
- 宿主项目必须提供哪些接口。
- 各平台差异说明：
  - `APP`
  - `H5`
  - `MP-WEIXIN`
- 登录成功后的默认行为是什么。
- 如何覆盖默认行为。

建议增加一节“接入前置条件”：

- 依赖的 uni_modules
- 依赖的全局对象
- 云函数配置
- 宿主项目接口约定

## 6. 一个务实的落地顺序

如果按投入产出比来排，我建议这样做：

1. 先修乱码、事件解绑、重复配置、演示残留代码。
2. 再抽离 `sampleApi`，把接口层改成适配器模式。
3. 再拆 `login-withoutpwd.vue` 的流程逻辑。
4. 最后补全 `jasper-login.vue` 和 `login-withpwd.vue`，让模块出口完整。

## 7. 总体评价

这个组件不是不能用，而是“能跑，但离稳定复用组件还有一段距离”。

它当前最值得优化的不是样式，而是工程化边界：

- 边界要清楚
- 配置要收敛
- 依赖要显式
- 流程要可替换
- 文档要能指导接入

如果你后续准备继续维护这个 uni_modules，我建议优先把它做成“宿主可注入 API 的登录框架”，而不是继续在页面文件里堆登录分支。
