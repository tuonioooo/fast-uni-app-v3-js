# 外部原型 UI -> 当前脚手架页面转换规范

## 1. 目标

本规范不是把原型“机械翻译”为 uni-app 代码，而是把外部设计转换为：

- 适配当前仓库目录结构的页面/组件代码
- 优先复用现有 `jasper-ui`、`uv-ui`、`uni-ui`、`jasper-login`
- 优先复用现有请求层、时间工具、加密工具、mock 机制
- 保持当前项目代码风格、样式变量、交互结构一致

最终目标是：生成的代码能直接放进当前仓库继续开发，而不是一份只能参考的 demo。

---

## 2. 当前仓库能力基线

在转换原型前，默认先按当前仓库已有能力落地：

### 2.1 技术栈

- `uni-app`
- `Vue 3`
- `Pinia`
- `Vite`
- `SCSS`

### 2.2 已有 UI 体系

优先级从高到低：

1. `jasper-ui`
2. `jasper-login`
3. `uv-ui`
4. `uni-ui`
5. `uni-app` 原生组件

### 2.3 已有工程能力

- 请求封装：`util/request/*`
- 常量插件：`common/plugins/constants.js`
- mock 能力：`mock/*` + `api/sample.js`
- 时间工具：`dayjs` + `uv-ui` 时间能力
- 加密能力：`crypto-js` + `jasper-ui` 工具封装
- 登录能力：`uni_modules/jasper-login`

### 2.4 已有页面组织方式

优先遵守当前仓库结构：

- 页面放在 `pages/*`
- API 放在 `api/*`
- 通用工具放在 `util/*`
- 状态放在 `stores/*`
- 可复用模块优先放 `uni_modules/*` 或页面内局部组件

---

## 3. 转换总原则

### 3.1 不是一比一复刻 DOM，而是适配当前脚手架

外部原型常见问题：

- 使用大量 `div/span`
- 使用 Web 专属布局
- 依赖 DOM/BOM
- 使用当前仓库没有的第三方组件
- 样式变量、颜色、间距与当前项目体系不一致

转换时必须优先做“项目适配”，不是“结构照抄”。

### 3.2 优先复用现有组件，不重复造轮子

只要现有组件库能覆盖，就不要手写重复 UI。

例如：

- 搜索栏优先 `uv-search`
- 表单优先 `uv-form`、`uv-form-item`、`uv-input`
- 弹窗优先 `uv-popup`
- 列表加载优先 `z-paging` 或 `uv-loadmore`
- 空状态优先 `uv-empty`
- 登录相关优先复用 `jasper-login`

### 3.3 优先复用现有工具，不直写散代码

例如：

- 接口请求优先走 `util/request`
- 业务 API 优先新建在 `api/*`
- mock 调试优先接入 `mock/*`
- 时间处理优先 `dayjs` 或已有工具
- 加密优先 `jasper-ui` 现有 helpers

### 3.4 优先贴合当前风格，不引入新体系

禁止在原型转换时随手引入：

- 新 UI 框架
- 新 CSS 方案
- 新请求库
- 新状态库
- 与当前仓库不一致的目录结构

---

## 4. 原型转换前必须确认的信息

### 4.1 页面目标

- 页面类型：列表页 / 表单页 / 详情页 / 首页 / 登录页 / 弹窗组件
- 放置位置：`pages/xxx/xxx.vue` 还是局部组件
- 是否需要注册到 `pages.json`

### 4.2 平台目标

- 微信小程序优先
- H5 优先
- 多端通用

这会直接影响：

- 页面滚动容器选择
- fixed/footer 处理
- 安全区处理
- 某些组件与交互实现方式

### 4.3 数据来源

- 纯静态展示
- mock 数据驱动
- 真实 API 驱动

如果是列表、表单、详情页，必须明确数据来源，不要只生成静态壳。

### 4.4 现有参考

优先读取并对齐：

- `uni.scss`
- `README.md`
- `pages.json`
- 一个已有页面，例如 `pages/index/index.vue`
- 相关模块文档，例如 `uni_modules/jasper-login/readme.md`

---

## 5. 组件映射规则

## 5.1 `jasper-ui` 优先

如果 `jasper-ui` 已有对应组件，优先使用，不再手写基础壳。

适用场景：

- 项目内统一卡片
- 统一标题区
- 统一导航栏
- 统一业务模块块状区
- 统一工具方法

要求：

- 先确认当前仓库中是否已有等价组件或 mixin
- 若名称不明确，先查看 `uni_modules/jasper-ui` 的实际实现或调用示例

## 5.2 `jasper-login` 专用于登录链路

适用场景：

- 登录页
- 验证码登录
- 密码登录
- 第三方登录
- 协议页跳转

要求：

- 登录相关页面不要另写一套新逻辑
- 优先复用 `jasper-login` 已有页面、store、service 注入机制

## 5.3 `uv-ui` 承担通用 UI

推荐映射：

| 原型元素 | 优先组件 |
|---|---|
| 顶部导航 | `uv-navbar` |
| 搜索栏 | `uv-search` |
| 按钮 | `uv-button` |
| 输入框 | `uv-input` |
| 表单 | `uv-form` / `uv-form-item` |
| 弹窗 | `uv-popup` / `uv-modal` |
| 标签 | `uv-tag` |
| 加载状态 | `uv-loading-icon` / `uv-loadmore` |
| 空状态 | `uv-empty` |
| 图片 | `uv-image` |
| 图标 | `uv-icon` |
| Tabs | `uv-tabs` |
| 轮播 | `uv-swiper` |

## 5.4 `uni-ui` 作补充

推荐场景：

- `uni-calendar`
- `uni-datetime-picker`
- `uni-collapse`
- `uni-countdown`
- `uni-notice-bar`

## 5.5 列表优先考虑 `z-paging`

对于以下页面，优先考虑 `z-paging`：

- 商品列表
- 订单列表
- 消息列表
- 任何需要下拉刷新 + 上拉加载更多的页面

不要每个列表页都手写一套分页逻辑。

## 6. 页面实现规则

### 6.1 模板层

必须使用 uni-app 组件语义：

- 容器：`view`
- 文本：`text`
- 图片：`image` 或 `uv-image`
- 滚动：`scroll-view` 或 `z-paging`

禁止直接保留 Web 语义标签：

- `div`
- `span`
- `section`
- `main`

### 6.2 脚本层

优先使用 `script setup`。

数据逻辑分层要求：

- 页面逻辑写在页面 `.vue`
- 业务接口写在 `api/*`
- 通用逻辑可抽到 `util/*` 或 hooks
- 不要把请求、数据转换、交互状态全部堆在模板页里

### 6.3 样式层

统一要求：

- 使用 `<style lang="scss" scoped>`
- 默认使用 `rpx`
- 优先复用 `uni.scss` 与项目变量
- 避免硬编码颜色、字号、圆角、阴影

如果原型色值很多，处理顺序应为：

1. 先找项目已有变量
2. 找不到时局部补充少量变量
3. 不直接把整张稿子的颜色全部硬编码进页面

### 6.4 页面注册

如果是新页面，必须补充：

- `pages.json` 片段
- 页面路径建议
- 是否属于 tabBar 页面

---

## 7. 数据与交互规则

### 7.1 请求必须走现有请求层

不要在页面里直接散写原始 `uni.request`。

推荐模式：

1. 在 `api/xxx.js` 定义接口
2. 页面中调用 `api/xxx.js`
3. 请求统一经过 `util/request`

### 7.2 mock 页面要接当前 mock 机制

如果用户只给原型，没有后端接口：

- 优先补充 `mock/data/*`
- 根据场景增加 `mock/mock.js` 或 `mock/index.js` 对应数据
- 页面调用 API 层，不直接写死假数据
- 图片资源优先使用 `https://picsum.photos/` 作为占位图来源，参考 `docs/guides/resources.md`
- 需要稳定复现同一张图时，优先使用 `https://picsum.photos/seed/<stable-seed>/<width>/<height>` 形式，避免列表刷新后图片随机跳变
- 原型落地阶段默认不要新增本地临时图片到页面目录，除非用户明确提供并要求使用固定素材

### 7.3 登录相关交互要与现有登录链路兼容

涉及以下场景时：

- 未登录跳转
- 登录后回跳
- 用户信息展示
- 退出登录

优先复用 `jasper-login` 已有能力，不另写第二套登录态。

## 8. 原型转换输出格式

每次输出都建议按下面顺序组织：

### 8.1 原型解析摘要

说明：

- 页面结构
- 使用到的组件
- 是否接 API
- 是否接 mock
- 平台注意事项

### 8.2 主页面代码

输出可直接落地的 `.vue` 页面代码。

### 8.3 子组件代码

如果页面复杂，可拆：

- 弹窗组件
- 卡片组件
- 筛选组件

### 8.4 API 文件

如果页面有接口，补充：

- `api/*.js`

### 8.5 `pages.json` 片段

新页面必须补。

### 8.6 TODO 清单

例如：

- 替换真实接口
- 补充字段类型
- 接入真实图片
- 补充权限判断

---

## 9. 当前仓库专属约束

### 9.1 登录页不要重复造

只要需求属于登录、验证码、密码登录、协议页，就先看 `jasper-login`，不要新写一套 UI 和状态。

### 9.2 列表页优先走统一列表范式

有分页时优先：

- `z-paging`
- 或统一 `loadmore` 模式

不要每页都各写一套分页状态机。

### 9.3 时间、加密、常量优先走项目已有能力

优先使用：

- `dayjs`
- `uni.$jasper`
- `common/plugins/constants.js`

### 9.4 页面视觉要贴近当前项目，不追求设计稿像素级复刻

当设计稿和当前组件体系冲突时：

- 先保结构
- 再保交互
- 最后才是视觉细节

能复用组件时，不要为了“更像图”而放弃组件体系。

## 10. 自检清单

生成代码前，至少自检以下内容：

- [ ] 是否优先复用了 `jasper-ui / jasper-login / uv-ui / uni-ui`
- [ ] 是否避免引入新的 UI 框架和请求库
- [ ] 是否使用 `view / text / scroll-view`
- [ ] 是否使用 `<style lang="scss" scoped>`
- [ ] 是否优先使用 `rpx`
- [ ] 是否补充了 `pages.json` 片段
- [ ] 是否把接口放到了 `api/*`
- [ ] 是否优先接入现有 `mock/*`
- [ ] mock 图片是否优先使用 `https://picsum.photos/`，并在列表场景使用稳定 `seed`
- [ ] 是否避免重复实现登录态
- [ ] 是否考虑了小程序/H5 多端兼容

---

## 11. 推荐执行顺序

1. 识别页面类型和平台目标
2. 读取当前仓库相关页面、样式、组件用法
3. 先做组件映射，不要急着写代码
4. 再按当前目录结构落页面、API、mock、`pages.json`
5. 最后检查是否真正复用了当前脚手架能力

这份 Skill 的核心不是“把图写出来”，而是“把外部原型转成当前项目真正能维护的代码”。
