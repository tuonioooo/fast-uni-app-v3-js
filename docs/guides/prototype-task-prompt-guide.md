# 原型图任务提问规范

这份文档用于规范在当前 `fast-uni-app-v3-js` 仓库中，如何把原型图、截图、设计稿或半成品页面交给 Codex 处理。

目标不是“把图画出来”，而是把原型转成当前项目里可继续联调、可继续维护的页面和业务链路。

## 1. 是否必须指定 skill

不是必须。

如果你只发原型图和目标要求，Codex 也可以根据任务内容自动判断。

但为了让执行更稳定、歧义更少，建议在请求里显式指定 skill。

常用的两个 skill：

- `ui-prototype-to-page`
  - 适合单页面、单组件、单弹窗落地
- `multi-role-ui-delivery`
  - 适合整条链路改造，例如 `列表 -> 详情 -> 表单 -> 成功弹窗`

## 2. 是否必须指定智能体

不是必须。

如果使用 `multi-role-ui-delivery`，会默认按下面顺序组织处理：

1. `requirements-analyst`
2. `frontend-engineer`
3. `refactor-engineer`
4. `tester`
5. `reporter`

只有在你明确想控制阶段时，才需要点名角色。

例如：

- “先只执行 `requirements-analyst`，先不要写代码”
- “这次直接按 `frontend-engineer` 实现，不用先出分析稿”

## 3. 每次至少要提供什么信息

建议至少提供下面 5 类信息：

- 原型图、截图、设计稿，或已有页面代码/页面目录
- 目标目录，例如 `pages/activity_detail`
- 是新建页面，还是改造已有页面
- 是否需要接入现有链路，例如从 `activity_list` 跳转过来
- 数据来源是 mock 还是真接口

如果信息更完整，效果会更稳定。推荐补充：

- 是否需要补 `pages.json`
- 是否需要检查未引用图片
- 是否要一起检查重复代码和可抽取逻辑
- 是否有固定图片素材，还是允许使用占位图

## 4. 当前仓库默认约定

默认按当前仓库规范落地：

- 页面放在 `pages/*`
- 优先复用现有 `uv-ui`、`jasper-ui`、请求封装和 mock 机制
- 新页面需要同步检查 `pages.json`
- 如果只是原型阶段，图片占位优先参考 `docs/guides/resources.md`
- 默认占位图使用 `https://picsum.photos/`
- 如果用户已提供本地素材，则优先使用本地素材

## 5. 推荐提问模板

### 5.1 单页面落地

适合单个列表页、详情页、表单页、弹窗页。

```text
使用 `ui-prototype-to-page`。

这是一个原型图/设计稿，请把它落到当前 uni-app 项目。

目标目录：
`pages/xxx`

要求：
1. 按当前仓库规范实现，优先复用现有组件和样式体系
2. 如需新页面，补充 `pages.json`
3. 如需数据，优先接 mock
4. 图片占位按 `docs/guides/resources.md`，使用 `https://picsum.photos/`
5. 检查页面目录下图片资源是否有未引用文件
6. 最后给我一个实现说明和验证结论
```

### 5.2 整条链路改造

适合 `列表 -> 详情 -> 表单 -> 成功弹窗` 这类任务。

```text
使用 `multi-role-ui-delivery`。

这是一个原型图/设计稿，请把它改造成当前项目里的完整业务链路。

目标链路：
`page_a -> page_b -> page_c -> modal_d`

目标目录：
- `pages/page_a`
- `pages/page_b`
- `pages/page_c`
- `pages/modal_d`

要求：
1. 先拆成功能需求和页面交互
2. 再按当前仓库规范完成实现
3. 页面之间要打通跳转、参数传递和返回逻辑
4. 如需 mock 数据，请一并补齐
5. 检查未引用图片、重复代码和可抽取逻辑
6. 最后做验证结论，明确是否已经形成闭环
```

### 5.3 只做需求分析

适合先看功能拆解，不急着落代码。

```text
使用 `multi-role-ui-delivery`，但这次先只执行 `requirements-analyst`。

这是原型图，请先不要写代码，只输出：
1. 页面目标
2. 页面区块结构
3. 交互流程
4. 字段设计
5. 路由和参数传递
6. 需要新增或复用的页面、弹窗、API、mock
7. 风险点和待确认项
```

### 5.4 已有页面优化改造

适合仓库里已经有初版页面。

```text
使用 `multi-role-ui-delivery`。

现有页面目录：
`pages/xxx`

参考原型图，对这个页面做改造。

要求：
1. 保留现有可用逻辑
2. 调整 UI 和交互，使其贴近原型
3. 检查是否有重复实现、无用图片、无用状态
4. 如涉及上下游页面，顺带打通交互
5. 最后告诉我改了什么、怎么验证、还有什么风险
```

## 6. 最推荐的日常发法

日常建议直接按下面格式发，信息量够、歧义小：

```text
使用 `multi-role-ui-delivery`。

这是活动详情页原型图。
请落到 `pages/activity_detail`，
并接到 `activity_list -> activity_detail -> activity_booking -> activity_success_modal` 这条链路里。
先用 mock，按当前仓库规范实现，最后检查未引用图片和交互闭环。
```

## 7. 什么时候用哪个 skill

可以按这个简单规则判断：

- 只是单页或单弹窗实现，用 `ui-prototype-to-page`
- 涉及多页跳转、参数流转、提交反馈、测试闭环，用 `multi-role-ui-delivery`
- 如果你不确定，也可以直接说“按当前仓库原型转页面流程处理”，Codex 会自行判断

## 8. 什么样的输入最容易得到好结果

下面这些输入方式，效果通常更好：

- 原型图命名清晰，能区分列表、详情、表单、弹窗
- 明确写出页面目录，不让 Codex 猜路径
- 明确写出是否要接现有页面
- 明确说明用 mock 还是真接口
- 明确说明图片是用本地素材，还是允许占位图
- 明确要求检查未引用图片和交互闭环

## 9. 建议避免的提法

下面这些说法容易导致反复确认或返工：

- “你看着做”
- “照着这个图搞一下”
- “随便放个目录吧”
- “逻辑你自己猜一下”

更稳妥的方式是至少补齐：

- 页面目录
- 是否新建
- 是否接现有链路
- 数据来源

## 10. 交付结果默认应该包含什么

一个合格的原型图任务交付，默认应该至少说明：

- 改了什么
- 页面或链路现在怎么走
- 是否补了 `pages.json`
- 是否补了 mock / API
- 是否检查了未引用图片
- 是否已经完成验证
- 还有哪些风险或待确认项

如果任务涉及业务闭环，最终最好明确写出类似结果：

- `activity_list -> activity_detail -> activity_booking -> activity_success_modal`

如果这条链还没打通，也应该明确说明，而不是默认当作已经完成。
