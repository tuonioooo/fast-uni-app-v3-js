# 当前快速开发工程待补充项

最后复核时间：2026-04-25

## 1. 当前工程已具备的基础能力

从现有仓库看，这个项目已经不是“空模板”，而是具备了一套可运行的基础底座，主要包括：

- 已集成 `uni-app + Vue3 + Pinia`
- 已接入 `uv-ui`、`jasper-ui`、`jasper-login`
- 已有统一请求封装、请求/响应拦截器
- 已有 `.env.*` 与 `.env.example`
- 已有 `vite.config.js`，并接入 `autoprefixer`、`postcss-px-to-viewport`
- 已有 `mock` 目录和 API 示例
- 已有 `ESLint`、`Prettier`、`husky`、`lint-staged`、`commitlint`
- 已有登录页面、登录 store、协议页、组件测试页

结论：当前项目已经可以作为“可运行的快速开发底座”，但距离“可长期复用的标准化业务脚手架”还差几项关键工程能力。

## 2. 优先建议补齐的内容

下面按优先级拆分。

### P0：建议优先补齐

说明：原 `2.1`、`2.2`、`2.3` 已完成，不再纳入“待补充项”清单。
其中 `vue-cli / CLI` 的命令脚本说明，将补充到 `jasper-login` 文档中。

#### 2.1 页面权限与路由访问约定

当前工程已有白名单配置，但更像是“能跑起来”的程度，还缺“模板级页面访问规范”。

建议补齐：

- 页面白名单约定
- 登录页禁止重复进入
- 未登录跳转登录页
- 登录后回跳原页面
- 页面是否需要登录的配置写法
- tabBar 页面与非 tabBar 页面跳转规范

补齐价值：

- 新业务页接入时有统一规则
- 减少页面跳转散落判断

预期交付物：

- 路由守卫实现
- 页面鉴权接入说明文档

#### 2.2 标准异常态页面与通用反馈组件

一个快速开发模板，不能只有“正常流程页”，还要把异常态也做成可复用能力。

建议至少补齐：

- 空状态页
- 无网络页
- 403 / 无权限页
- 404 / 页面不存在页
- 通用加载中骨架或 loading 组件
- 全局错误提示与二次封装 toast / modal

补齐价值：

- 新业务不用重复写状态页
- 页面体验完整度明显提升

预期交付物：

- `pages/status/*` 或 `components/base-status/*`
- 使用规范文档

### P1：建议尽快补齐

#### 2.3 基础业务模板页

当前有示例页，但还缺真正适合复制新业务的页面模板。

建议补齐：

- 列表页模板
- 详情页模板
- 表单页模板
- 搜索筛选页模板
- WebView 页模板
- 上传示例页

补齐价值：

- 新页面可以直接复制改业务
- 团队页面结构更统一

预期交付物：

- `pages/template/*`
- 页面开发约定文档

#### 2.4 通用能力封装

建议把高频业务能力提前沉淀，不要等每个项目再重复做。

建议补齐：

- 本地存储封装
- 文件上传封装
- 文件下载/预览封装
- 分页查询参数封装
- 表单提交防重复封装
- 平台判断与设备信息工具
- WebView 通讯封装

补齐价值：

- 大多数业务项目都会直接用到
- 能显著减少页面层样板代码

预期交付物：

- `utils/storage.js`
- `utils/upload.js`
- `utils/download.js`
- `composables` 或 `hooks` 目录下的通用 hooks

#### 2.5 统一配置中心

目前环境变量已有，但模板工程还可以进一步收口配置，减少散落常量。

建议补齐：

- 接口地址配置
- 静态资源地址配置
- 登录页路径配置
- token key 配置
- 业务成功码配置
- 调试开关配置

补齐价值：

- 配置项集中、可替换、可审计
- 多项目复用时更容易改造

预期交付物：

- `config/` 目录
- 配置说明文档

#### 2.6 API 分层规范

当前已有 `api/sample.js`，但还没有形成模板级的模块化 API 组织方式。

建议补齐：

- `api/modules/*` 目录结构
- 请求参数与响应数据的约定
- 业务成功码、错误码、提示策略统一
- 接口命名规范

补齐价值：

- 后期接口量大时不容易失控
- 便于多人并行开发

预期交付物：

- API 目录规范
- 接口开发说明文档

### P2：可按阶段补齐

#### 2.7 自动化测试

JS 模板不一定一开始就上完整测试体系，但至少可以从工具函数和请求层开始。

建议补齐：

- 工具函数单测
- 请求层单测
- 登录流程关键路径测试

#### 2.8 CI/CD

建议后续补齐：

- PR 自动 lint
- 自动格式校验
- 自动构建检查
- 发布说明或变更日志自动化

#### 2.9 TypeScript 版本分支

当前项目是 JS 模板，适合上手快；如果后续要作为长期对外模板维护，建议考虑补一套 TS 版本。

建议方向：

- 保留当前 JS 版本
- 新增一套 `fast-uni-app-v3-ts` 或 TS 分支

#### 2.10 国际化与主题能力

如果后续会服务多个项目，建议中长期补齐：

- i18n 国际化
- 主题色切换
- 品牌配置切换
- tabBar 主题方案切换

## 3. 推荐的落地顺序

如果只打算做一轮完善，建议按下面顺序推进：

1. 补齐页面权限访问约定与登录失效处理
2. 补齐标准异常态页面和基础模板页
3. 再整理通用能力封装与配置中心
4. 后续按需要补充测试、CI/CD 与 TS 分支

## 4. 一句话总结

当前工程已经完成了“登录能力 + UI 基础能力 + 请求基础能力”的第一阶段建设；接下来最值得补的，不是继续堆组件，而是补齐“脚手架级工程能力”，尤其是：

- 页面鉴权规范
- 异常态页面
- 列表/详情/表单等业务模板页

这些内容补齐后，这个项目就更接近一套真正可长期复用的 uni-app 快速开发模板。
