# Codex Agents Workflow

本仓库采用 `AGENTS.md + .codex/agents + .codex/skills` 的协作方式组织多角色智能体。

## 全局原则

- 先分析，再实现，再收口，不跳步骤。
- UI 原型类需求优先复用 `.codex/skills/ui-prototype-to-page/SKILL.md`。
- 页面改造必须优先复用当前仓库已有的 `uv-ui`、`jasper-ui`、请求封装、mock 数据机制与页面结构。
- 新页面或新链路必须同步补齐 `pages.json`、必要的 API、mock、测试入口与白名单。
- 输出必须说明：改了什么、涉及哪些文件、怎么验证、剩余风险是什么。

## 推荐角色分工

### 1. requirements-analyst

负责把 UI、原型图、截图、静态页面转成可执行的功能需求。

- 拆页面结构
- 梳理交互与状态
- 定义字段、接口、提交流程
- 识别与现有页面/模块的交互关系

### 2. frontend-engineer

负责基于既定需求在当前 uni-app 脚手架里完成页面实现。

- 优先复用组件库和已有样式体系
- 补齐页面、API、mock、路由、测试入口
- 保证页面在当前项目中可直接接入

### 3. refactor-engineer

负责在不改变功能目标的前提下收敛复杂实现，提升可维护性。

- 消除原型切图页、重复代码和硬编码
- 提取共享逻辑、状态和文案常量
- 保持行为一致，降低后续迭代成本

### 4. tester

负责对已实现页面和链路做验证设计、回归检查和风险汇总。

- 检查页面入口、跳转链路、表单校验、弹窗反馈
- 校验 `pages.json`、mock、API、白名单是否同步
- 优先输出真实问题、缺口和回归风险
- 不把“看过代码”当成“完成验证”

## 标准执行顺序

1. `requirements-analyst` 先把原型翻译成结构、字段、交互、边界条件。
2. `frontend-engineer` 按仓库规范落页面与链路。
3. `refactor-engineer` 对页面与逻辑做收敛整理，避免临时实现遗留。
4. `tester` 做链路验证、问题枚举和交付前检查。

## 原型改造专项要求

- 来源是图片/原型时，先明确“视觉结构”和“功能结构”，不要直接照搬切图布局。
- 列表、详情、表单、弹窗之间的跳转链路必须打通，而不是只做单页展示。
- mock 图片默认优先使用 `https://picsum.photos/`，但如果用户明确给了固定素材，则优先使用本地素材。
- 成功弹窗、表单页、详情页这类中间状态页面，优先抽成可复用组件或共享数据链路。
- 验证时至少覆盖：页面入口、参数传递、空态、提交成功、提交失败或校验失败。

## 交付模板

- 需求摘要：页面目标、关键交互、数据来源
- 实现结果：新增/修改页面、接口、mock、配置
- 验证方式：如何从入口点到结果页走通
- 风险说明：未接真实接口、未跑真机、待产品确认项
